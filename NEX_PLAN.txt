STEP 3 — Reasoning Transparency in API responses
  Goal: Every /api/chat response includes a readable belief chain
        showing HOW NEX arrived at its answer, not just WHAT it said.
  Files: nex_api.py, nex_reason.py
  Output: Add "reasoning_chain" field to chat response JSON
          [{belief, topic, confidence, role: "supporting"|"opposing"}]
  Value: Enterprise buyers need auditability. This is the thing that
         separates NEX from a black-box LLM.

STEP 4 — Topic Alignment + Belief Quality Metrics in /api/stats
  Goal: /api/stats returns metrics that prove NEX improves over time.
  Files: nex_api.py, nex_belief_quality.py
  Add to stats response:
    - topic_alignment_pct (currently ~40.3%, should climb)
    - avg_confidence_trend (weekly)
    - belief_quality_score (from nex_belief_quality.py)
    - synthesis_count (from scheduler)
  Value: The "gets smarter over time" claim needs to be measurable
         and visible in the API. This is the sales metric.

STEP 5 — Tier Enforcement (Personal/Pro/Enterprise behave differently)
  Goal: API key tiers actually enforce different capabilities.
  Files: nex_api.py, ~/.config/nex/api_keys.json
  Personal ($199):
    - 100 requests/day cap
    - No webhooks
    - No GDPR export
    - Single session only
  Professional ($2,500/mo):
    - 1,000 requests/day
    - Webhooks enabled
    - Full audit trail
    - Up to 10 sessions
  Enterprise ($15,000/mo):
    - Unlimited requests
    - All features
    - reasoning_chain enabled
    - Domain activation
  Value: Required before any real commercial sales. Currently all
         keys have enterprise-level access regardless of tier.

STEP 6 — Session Persistence Across Restarts
  Goal: Sessions survive API restart. Currently in-memory only.
  Files: nex_api.py
  Method: Write sessions to ~/.config/nex/sessions.json on update,
          load on startup. Same pattern as api_keys.json.
  Value: Enterprise requirement. Buyers cannot lose conversation
         context every time the server restarts.

STEP 7 — nex_v72 Singleton Resolution
  Goal: Remove "[loop_wiring] Could not load v72 singleton" warning.
  Files: nex_loop_wiring.py
  Method: Read actual nex_loop_wiring.py, find what it tries to
          import from nex.nex_v72, either create the stub or
          remove the dependency cleanly.
  Value: Professional — no warnings in production logs.

STEP 8 — QLoRA Fine-Tune Pipeline
  Goal: Fine-tune Qwen2.5-3B on NEX's own belief corpus.
  Prerequisite: Domain corpus quality — need 200+ beliefs per domain
                with confidence > 0.7 before training is worthwhile.
  Files: New — nex_finetune.py
  Target hardware: RunPod A100 (see TRAINING_PROTOCOL.md)
  Protocol: Kill llama-server first, HSA_OVERRIDE_GFX_VERSION=10.3.0,
            HIP_VISIBLE_DEVICES=0, dtype=torch.float16,
            device_map={"":0}, t.join() after t.start(),
            no BitsAndBytes, PyTorch 2.5.1+rocm6.2
  Value: This is the moat. A model trained on NEX's own beliefs
         cannot be replicated. Pushes NEX from "interesting agent"
         to "proprietary intelligence asset."

STEP 9 — Multi-Tenant Key Management UI
  Goal: Web UI for managing API keys, viewing usage, upgrading tier.
  Files: New page in docs/ or new Flask blueprint in nex_api.py
  Value: Self-serve commercial onboarding. Required to scale beyond
         manual key creation.

STEP 10 — Webhook Delivery Dashboard Panel
  Goal: Show webhook delivery stats in nex_dashboard.py
  Files: nex_dashboard.py, nex_api.py
  Add: Delivery success rate, retry count, last fired timestamp
  Value: Enterprise buyers need to monitor their integrations.
