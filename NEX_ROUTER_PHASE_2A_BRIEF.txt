Phase 2A: Response Router — Tiered Inference Infrastructure
================================================================================

Context: Phase 1D shipped as commit 0cf70a5. Substrate hygiene loop is
closed. C1 leak under freeze is 15.3%, reproducibly traceable to LoRA
weight contamination on 3 specific seed patterns. FT#11 is blocked on
RunPod credits.

This phase is NOT about fixing C1 at the root (that's FT#11) and NOT
about capability progress (that's Stage B tool use, later). It's about
formalizing the implicit PATH 1 / PATH 2 split into an explicit tiered
router with measurable behavior.


--------------------------------------------------------------------------------
WHY THIS BRIEF EXISTS
--------------------------------------------------------------------------------

Current nex_respond_v2.py routing logic:
  - call_llm runs retrieval, then tries _build_reply (PATH 1)
  - if PATH 1 returns >20 chars, use it
  - otherwise fall through to PATH 2 (LLM call)

The routing is implicit. Nobody looks at what fraction of live traffic
hits each path, how latency varies, or which queries deserve which
path. The PATH 1 renderer was a happy accident — Phase 1B treated it
as a bug to bypass, then Phase 1C found it was doing real work.

Hypothesis (testable, not asserted): the majority of NEX's queries
are retrievable compositions that don't need the LLM at all. If true:
- most LLM calls are paying ~2s + GPU load to produce template-plus-
  connectives that Python can do in microseconds
- most LLM calls are the source of the bridge:X↔Y contamination that
  Phase 1D traced to LoRA weights
- routing aggressively away from the LLM reduces contamination
  exposure proportionally, while we wait on FT#11

Phase 2A builds the router, measures the hypothesis, and — if
confirmed — deploys it as infrastructure for all downstream work
(Stage B tool use, Stage C planning, FT#11 validation).


--------------------------------------------------------------------------------
SUCCESS CRITERIA
--------------------------------------------------------------------------------

R1: Router + Tier 0 composer + route_decisions table exist. Unit tests
    pass. No production code touched yet.

R2: Blind test — 20 queries, response from Tier 0 and Tier 2, shuffled,
    Jon's accuracy at identifying which is which is reported honestly.
    The *number* is the deliverable, not a pass/fail. If Jon is 95%
    accurate, Tier 0 isn't ready. If Jon is 55% accurate (near chance),
    Tier 0 is doing real work.

R3: Latency test — 50 queries through the router (flag on) vs 50 through
    current path (flag off). Mean latency delta reported.

R4: C1 test — 50 queries through router vs 50 through current path.
    C1 rate compared. Predicted drop: proportional to Tier 0 route
    rate. Actual drop reported.

R5: No regression test — R6 probe (all 6 queries including adversarial
    identity probe) runs through the router. Every probe routes to
    Tier 2. Responses are byte-identical or near-identical to Phase 1D
    Tier 2 baseline. If any R6 probe routes to Tier 0 or 1, that's
    a bug.

None of R1-R5 is pass/fail. They are measurements. The deliverable is
the data + a recommendation (promote flag default, keep flag off, or
revise).

TASK 2 — Create nex_tier0_composer.py (auto-accept)
--------------------------------------------------------------------------------

Per sketch Section 4. Key requirements:

1. Every belief passes through _sanitize_belief (imported from
   nex.nex_respond_v2) before composition. No exceptions.

2. Output length bounded: minimum 40 chars (to avoid trivial outputs),
   maximum 400 chars (to keep Tier 0 concise by design).

3. Intent-aware templates:
   - self_inquiry → _SELF_OPENERS + body
   - factual → declarative concat, no opener
   - general → _GENERAL_OPENERS + body with optional connective

4. Deterministic given (beliefs, intent, query): use hash-based
   selection from opener/connective lists, not random.choice. This
   is important for the blind test — same input must produce same
   output across runs.

5. Connective vocabularies per sketch Section 4. Hand-curated. Small.

Unit tests:
- Given 3 high-confidence beliefs and self_inquiry intent, produces
  a valid-looking response.
- Given a belief containing "bridge:X↔Y", the composed output does
  NOT contain "bridge:X↔Y" (sanitizer fires).
- Same input always produces same output (determinism).
- Output length is in [40, 400].

Auto-accept. No production touch.

TASK 4 — Wire router into generate_reply (REVIEW-GATED)
--------------------------------------------------------------------------------

PAUSE. Show Jon the proposed diff before writing.

The goal: router sits between retrieval+intent and the current PATH 1/
PATH 2 branch. Gated by NEX_ROUTER env flag. Flag off = zero change
from current behavior.

Proposed shape:

    # in generate_reply, after retrieval and intent classification

    if os.environ.get("NEX_ROUTER") == "1":
        from nex_response_router import route, RouteInput, BeliefHit
        hits = [BeliefHit(content=b.content, confidence=b.confidence,
                          topic=b.topic, tfidf_score=b.score)
                for b in retrieved_beliefs]
        ri = RouteInput(query=query, beliefs=hits, intent=intent,
                        source=os.environ.get("NEX_ROUTER_SOURCE", "live"))
        decision = route(ri)

        if decision.tier == 0:
            response = decision.composed_text
            log_decision(ri, decision, response, latency_ms=0)
            return response

        if decision.tier == 1:
            # Light LLM config
            response = call_llm(
                system=SYSTEM_PROMPT_LIGHT,
                user=build_light_prompt(query, hits[:3]),
                max_tokens=150, temperature=0.2,
            )
            log_decision(ri, decision, response, latency_ms=...)
            return response

        # Tier 2 falls through to existing PATH 2 logic
        # (and log_decision is called there)

    # Existing logic unchanged when flag is off
    ...

Constraints:

- Tier 0 response still passes through post_filter (sanitization,

- SYSTEM_PROMPT_LIGHT is a new shorter prompt for Tier 1. Draft it
