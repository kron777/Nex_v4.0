Phase 1D: Experiment Freeze + Corpus Hygiene for LoRA Retrain
================================================================================

Phase 1C shipped as commit ad5395d. C2 and C3 passed. C1 failed at 22.4%
because bridge:X↔Y syntax is baked into the LoRA weights (nex_lora_f16.gguf),
not injected from beliefs. Sanitizer works at the serialization layer; the
LLM generates the contamination from training.

Phase 1D does NOT attempt to fix C1 with a post-generation output sanitizer.
That would be cosmetic — the LLM would still be thinking in graph syntax;
we'd just wipe it before display. Phase 1D prepares the ground for the real
fix (FT#11 LoRA retrain on sanitized corpus) and fixes two infrastructure
issues Phase 1C exposed that will bite any future characterization.


--------------------------------------------------------------------------------
WHY THIS BRIEF EXISTS
--------------------------------------------------------------------------------

Phase 1C surfaced three problems beyond C1:

1. CORPUS DRIFT DURING EXPERIMENTS. Run 1 of each N=3 triplet differed
   from runs 2-3 because the live brain ingested beliefs between the
   baseline and run 1. N=3 wasn't really N=3 on the same system.

2. DATABASE LOCK CONTENTION. HUD shows persistent sqlite3.OperationalError:
   database is locked during normal operation. path2_log writes occasionally
   fail silently because of busy_timeout exhaustion. Same problem will
   affect any longer-running characterization.

3. LORA TRAINING CORPUS IS CONTAMINATED. The current LoRA (nex_lora_f16.gguf)
   was trained on beliefs that included bridge:X↔Y, conf=X.XX, edge labels.
   FT#11 when it runs needs a cleaned corpus. If the corpus is cleaned
   wrong, FT#11 will produce another contaminated model.

Phase 1D addresses 1 and 2 directly, and produces the cleaned corpus
export for FT#11 so when RunPod credits land, the retrain is unblocked
and has a known-clean training set.

    is > 6, break into two batches for review.

  sudo systemctl restart nex-brain

TASK 3 — path2_log write reliability (auto-accept for diagnostics,
         REVIEW-GATED if changes are needed)

Diagnostic first. Synthetic load test:

                     belief_count=0, system_prompt='', response_raw='test',

  - Increase busy_timeout from 300 to 3000 (3 seconds).
